# Contributing to PyTNL

Thanks for your interest in contributing to **PyTNL**!

PyTNL provides Python bindings for the [Template Numerical Library (TNL)](https://tnl-project.org/) —
a modern C++ library for efficient numerical solvers and HPC algorithms. PyTNL enables Python-driven
workflows while keeping performance-critical kernels in compiled backends.

- **Documentation:** https://tnl-project.gitlab.io/pytnl/
- **Repository:** https://gitlab.com/tnl-project/pytnl
- **Issues:** https://gitlab.com/tnl-project/pytnl/-/issues
- **License:** MIT (see `LICENSE`)

> **Supported Python versions:** PyTNL supports **Python 3.12+** (`requires-python = ">=3.12"`).
> Please use Python 3.12 or newer for development.

---

## AI Assistance Notice (Important)

If you are using any kind of AI assistance to contribute to PyTNL, it must be disclosed in the pull request.

If you are using any kind of AI assistance while contributing to PyTNL, this must be disclosed in the pull request,
along with the extent to which AI assistance was used (e.g. docs only vs. code generation). If PR responses are being
generated by an AI, disclose that as well. As a small exception, trivial tab-completion doesn't need to be disclosed,
so long as it is limited to single keywords or short phrases.

Example disclosures:

- This PR was written primarily by Claude Code.
- I consulted ChatGPT to understand the codebase but the solution was fully authored manually by myself.

Failure to disclose this makes it difficult to determine how much scrutiny to apply to the contribution.
Please be respectful to maintainers and disclose AI assistance.

## What we accept

We welcome contributions such as:

- Bug fixes (preferably with a regression test)
- Documentation improvements (clarity, examples, tutorials)
- Performance improvements (include measurements when possible)
- Typing improvements (type hints, stubs, narrowing `Any`)
- Build / packaging improvements (CMake, scikit-build-core, wheels)
- CI improvements
- New features (please discuss first if non-trivial)

We generally avoid:

- Large rewrites without prior discussion
- PRs that mix many unrelated changes (refactor + feature + formatting in one)
- Drive-by formatting changes across many files (unless explicitly needed)

If you want to implement a larger change, start by opening an issue describing:
the problem, your proposed solution, and any alternatives considered.

---

## Other tips and tricks (PROP-style)

These habits make PRs easier to review and more likely to be merged.

- **Keep PRs small and focused.** One PR = one problem. Split big work into multiple PRs.
- **Prefer "fix + tests" together.** Add regression tests for bug fixes.
- **Avoid unrelated formatting churn.** It makes reviews harder and hides real changes.
- **Use `git add -p`.** Stage only the relevant hunks so commits stay clean.
- **Explain the why.** In PRs/commits, include motivation and context, not only what changed.

## Where to ask questions

- **Bug reports / feature requests:** open an issue
  https://gitlab.com/tnl-project/pytnl/-/issues
- **PR discussion:** use the PR comments and keep the conversation tied to code changes.
- **General questions:** if the project uses GitLab Discussions or another channel, use that (otherwise use issues).

## Reporting issues

A good bug report includes:

- **Steps to reproduce** (minimal script or minimal input)
- **Expected behavior** vs **actual behavior**
- **Environment**
  - OS (Linux/macOS/Windows)
  - Python version (3.12+)
  - PyTNL version
  - toolchain details (compiler, CUDA, MPI) if relevant
- full logs / tracebacks

Try to keep reproductions minimal:
- a 20–50 line script is perfect, or
- a new unit test you can paste into `tests/`.

Feature requests should include:
- the use case ("what are you trying to do?"),
- proposed API shape (Python-facing),
- any relevant TNL component,
- alternatives you considered.

## Development setup

### Requirements

- Python **3.12+**
- `git`
- Recommended: `just`
- A working C++ toolchain (PyTNL builds native extensions)

Build note: PyTNL builds C++ extension modules via `scikit-build-core`, so you may also
need a working CMake toolchain (e.g. CMake + Ninja + a compiler) depending on your platform.

For GPU/MPI work you may also need:
- CUDA toolkit (CUDA builds/tests)
- MPI + `mpirun` (for MPI tests)

### Clone

```bash
git clone <your-fork-url>
cd <repo>
```

## Install dev dependencies

PyTNL defines optional dependency groups in `pyproject.toml`. For development,
use the `dev` extras (and optionally `dev-cuda`).

Recommended (via `just`):

```bash
just install
```

This installs in editable mode with dev extras. In plain pip terms this is roughly:

```bash
pip install --no-build-isolation -ve ".[dev,dev-cuda]"
```

Note: `dev-cuda` currently uses `cupy` (build from source) because wheels may not exist
for the newest Python versions.

## Git LFS

### Setup (optional)

If the repository uses Git LFS, install it and pull LFS objects:

```bash
git lfs install
git lfs pull
```

### Troubleshooting: files contain only “metadata” / pointer text

If you open a file and it starts with:

- `version https://git-lfs.github.com/spec/v1`
- `oid sha256:...`
- `size ...`

then you’re seeing a **Git LFS pointer**, not the real file content.

Fix:

```bash
git lfs install
git lfs pull
```

If that doesn’t fix it, try:

```bash
git lfs fetch --all
git lfs checkout
```

Common causes:
- LFS is not installed or not enabled locally
- LFS objects were not downloaded yet
- CI disables automatic LFS checkout (then `git lfs pull` must be explicit)

## Development workflow

### Branching model

Create a feature branch from the default branch (e.g. `main`) and keep it focused:

```bash
git checkout -b feature/short-description
```

### Rebase-based workflow (recommended)

We prefer a clean, easy-to-review history:

- Keep your branch up to date by rebasing onto the latest default branch.
- Before requesting review, consider squashing “fix typo”, “oops”, “wip” commits.

Typical workflow:

```bash
git fetch origin
git rebase origin/main
```

Interactive rebase to clean up commits:

```bash
git rebase -i origin/main
```

Notes:
- If you already opened a PR and reviewers started reviewing, avoid rewriting history unless necessary.
- If you do rewrite history (force-push), mention it in the PR so reviewers know they need to re-review.

## Commit messages

Write commit messages that are easy to scan in the log.

Guidelines:
- Subject line: **imperative mood**, no trailing period.
- Keep it short (ideally ~50 characters).
- If needed, add a blank line and a body with context (wrap at ~72 chars).

Examples:
- `Fix DLPack writable flag handling`
- `Add MPI regression test for ...`
- `Improve docs for installation on Linux`

If the commit is non-trivial, the body should include:
- motivation (why),
- approach (what you did),
- tradeoffs/limitations (if any).

## Development commands (Just)

PyTNL uses `just` (see `justfile`) as the primary entry point for development tasks.
Using `just` is strongly recommended because it keeps your workflow consistent with CI.

To list all available recipes:

```bash
just --list
```

### The “run everything” command (recommended)

Before opening a PR, run:

```bash
just check
just test
```

`just check` runs a suite of fast checks expected by CI/review:

- `check-format` — formatting check (Ruff formatter, diff-only)
- `check-code` — linting (Ruff)
- `check-typing` — type checking (basedpyright + mypy)
- `check-typos` — spelling/typos (`typos`)
- `check-recipes` — shellcheck validation for `justfile` recipes

### Formatting

Check formatting (no changes applied; shows what would change):

```bash
just check-format
```

Apply formatting (writes changes):

```bash
ruff format
```

### Linting

Run Ruff lint checks:

```bash
just check-code
```

### Type checking

Run type checks (basedpyright + mypy):

```bash
just check-typing
```

### Typos

Run the typos checker:

```bash
just check-typos
```

### Tests

Run the test suite:

```bash
just test
```

### Build artifacts

Build everything:

```bash
just build
```

Build wheel only:

```bash
just build-wheel
```

Build sdist only:

```bash
just build-sdist
```

## Code style and quality checks (Ruff)

This project uses **Ruff** for both formatting and linting. Ruff is the source of truth
for code style: do not apply ad-hoc formatting rules.

Practical guidance:
- Run `ruff format` before committing.
- Run `just check-code` before pushing.
- Avoid unrelated formatting changes in logic PRs.

If you want to run Ruff directly:

```bash
ruff format.
ruff check .
```

---

## Type checking (basedpyright + mypy)

PyTNL runs both **basedpyright** and **mypy**.

Run locally:

```bash
just check-typing
```

Guidelines:
- Add type hints for new public functions/classes.
- Prefer precise types at public boundaries (public API).
- If you must use `# type: ignore[...]`, keep it narrow and add a short explanation comment when non-obvious.

## Testing (pytest)

Run tests:

```bash
just test
```

Pytest is configured to:
- use strict markers,
- run in parallel (`-n auto`),
- exclude CUDA tests by default (`-m 'not cuda'`).

### When to add tests

Add or update tests when:
- fixing a bug (add a regression test)
- adding a feature
- changing behavior
- changing public API

### Markers: `cuda` and `mpi`

Some tests require special environments:

- `@pytest.mark.cuda` for tests that require CUDA execution
- `@pytest.mark.mpi` for tests that require `mpi4py` and running via `mpirun`

Keep CPU-only tests unmarked and deterministic.

---

## CUDA and MPI

### CUDA

CUDA-specific tests should be marked with `cuda`.
Because CUDA availability varies by machine/CI runner, CUDA tests should:
- fail with clear messages when CUDA is unavailable, or
- skip appropriately when the environment is not configured.

Example (run CUDA-marked tests):

```bash
pytest -m cuda
```

### MPI

MPI tests should be marked with `mpi` and documented in the test file:
- how to run them locally (expected `mpirun` usage),
- what assumptions they make about environment and ranks.

Example (run MPI-marked tests with 4 ranks):

```bash
mpirun -n 4 pytest -m mpi
```

## Documentation contributions

Documentation (including installation instructions) is hosted on GitLab Pages:
https://tnl-project.gitlab.io/pytnl/

Docs PRs are welcome. Good documentation contributions:
- include runnable examples,
- explain why and when to use a feature (not just how),
- avoid stale instructions (especially for Python 3.12+ environments).

If you add or change an API, update docs in the same PR when possible.

## Quick checklist (before PR)

If you changed code:

```bash
ruff format
just check
just test
```

If you changed typing-heavy code (public APIs, wrappers, annotations), also run:

```bash
just check-typing
```

If you changed docs only, at minimum run:

```bash
just check-typos
```

Always remember:
- update tests/docs for user-facing changes,
- disclose AI assistance in the PR (if used).

## Pull request guidelines

### What makes a great PR

A great PR is:
- scoped to one change,
- easy to understand,
- tested,
- documented (if user-facing),
- aligned with project tooling (`just`, Ruff, type checks).

### PR checklist (before opening)

Run:

```bash
just check
just test
```

Ensure:
- formatting is applied (`ruff format` if needed),
- lint/type checks pass,
- tests pass,
- new behavior is covered by tests,
- docs updated if user-facing behavior changed,
- AI assistance is disclosed (if used).

### PR description template (recommended)

Use this template in the PR description:

```md
### Summary
### Motivation
### Testing
### Breaking changes
### AI disclosure
```

Example:

- Summary: Fix XYZ by handling ABC correctly.
- Motivation: Prevents crash and matches expected semantics.
- Testing: `just check`, `just test`
- Breaking changes: None
- AI disclosure: None

### Avoid hidden breaking changes

If you change behavior or API:
- call it out explicitly,
- add tests,
- update docs,
- consider adding a deprecation path if appropriate.

## Review etiquette

- Respond to review feedback respectfully and with context.
- Batch related changes (avoid many micro-commits unless requested).
- Prefer resolving conversations in the UI rather than posting "fixed" comments.
- If you disagree with a suggestion, explain why and propose an alternative.
